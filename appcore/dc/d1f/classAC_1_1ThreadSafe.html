Detailed Description

Provides a basic framework for writing thread-safe objects.

Note:
It is left to the descended classes to lock themselves when necessary.
Unlike the standard pthread interface, one unlock doesn't clear the object for writing, but the number of unlocks must exactly match the number of locks. Also note that this does not apply to rwLocks, as only one thread can rationally modify the object at a time.
Constructors and destructors will never be thread-safe, so don't waste your time trying.
Neither of the two write-lock functions are technically const, but in the setting they are used they often must be considered so by the child class. Without this slight oversimplification, you would not be able to use locks within children's const member functions.
